Documentation

Muro builds the CSS layer of a product from one clean `.mss` entry.

This site is intentionally simple: one compiler, one output file, and a documentation layout that shows the workflow directly. Muro handles variables, nesting, modules, source maps, utility generation, asset rewriting, theme baking, and watch mode without handing control to a stack of extra CSS tools.

Quick start 
npm i -D muro-css

{
  "scripts": {
    "muro": "muro src/app.mss public/app.css",
    "muro:watch": "muro src/app.mss public/app.css --watch"
  }
}

npm run muro
npm run muro:watch
Feature set

One compiler handling the work teams usually spread across several tools.

The focus is not novelty for its own sake. Muro is useful when a project needs expressive authoring syntax, modern browser handling, and production output that still maps back to source cleanly.

Shared utility generation

Repeated declaration groups are detected and rewritten into generated `.muro-u-*` utilities and custom properties.

Theme baking

Use `light(...) dark(...)` in authored rules and let Muro emit the matching dark-mode output for you.

Muro Guard

Lock brand-critical declarations so invalid overrides stop at compile time instead of leaking into production.

Asset rewriting

Raster asset URLs can be converted to generated `.webp` files during the build when a converter is available.

Container math

Use `cw`, `ch`, `ci`, `cb`, `cmin`, and `cmax` helpers for container-aware sizing expressions.

Source maps and watch

Keep browser inspection tied to original source files and stay in an edit loop with one watch command.

Usage

Use one entry file to compose a whole site stylesheet.

This website follows the same pattern it documents: `src/app.mss` imports tokens, layout, and component partials, then compiles into one `muro.css` file beside the HTML shell.

`src/app.mss` 
@import "./_base";
@import "./_layout";
@import "./_components";

.site-shell {
  color: light(#121315) dark(#f4f4f1);
}
Commands 
npm run muro
npm run muro:watch

npx muro src/app.mss muro.css
npx muro src/app.mss muro.css --sourcemap
npx muro src/app.mss muro.css --watch
Site build shape: One entry, one output file, and no separate post-processing pass to bolt on themes, maps, purge, or utility extraction.
Docs snapshot

Start with the commands and imports developers look for first.

The surface area is intentionally legible. Built-ins are importable, comments are flexible, and the migration path still accepts old `.x2s` input while teams move to `.mss`.

Built-in imports

@import "muro:reset";
@import "muro:grid";
@import "muro:typography";

.frame {
  @include muro-container(80rem, 1.5rem);
}

Language notes

  • `.mss` is the primary format, with `.x2s` kept as a bridge release input.
  • `//`, leading `#`, and `** ... **` comments are all supported.
  • `muro:all` can pull the built-in reset, grid, and typography modules together.

Important flags

  • `--watch` for iterative development loops.
  • `--sourcemap` or `--inline-sourcemap` for browser inspection.
  • `--purge`, `--asset-dir`, and `--no-assets` for production control.
Install

Install once, then keep the workflow branded around `muro`.

Use the package as a dev dependency in product repos. The package name is `muro-css`; the command developers actually run remains `muro`.

Install path 
npm i -D muro-css

{
  "scripts": {
    "muro": "muro src/app.mss public/app.css",
    "muro:watch": "muro src/app.mss public/app.css --watch"
  }
}

npm run muro

Migration bridge

The current release still compiles legacy `.x2s` files and keeps the `x2s` binary alias alive for one transition window.

.mss muro muro-css
